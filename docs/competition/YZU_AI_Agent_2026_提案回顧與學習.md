## 2. 入圍作品的共同模式（我們該學的）

### A. 題目「在晶圓廠敘事裡」，不是「周邊」

入圍題目幾乎全部直接對準：

- 良率、測試、設備維護  
- ESG / 永續報告  
- 產業情報、輿情、戰略情境  
- 半導體 **產業內** 資料與決策

我們的 Fab-Truth 定位是 **「半導體產業鏈下游與周邊硬體（治具、開發板、IoT）」** — 邏輯上合理，但對 **欣銓書院評審** 可能讀起來像：

- 通用硬體 NPI 工具  
- 離「半導體 AI Agent 競賽」主視覺多一步  

### B. Agent **研究語彙** 很明顯

入圍標題常出現：

- **Agentic** 迴圈（目標—規劃—執行—驗證）  
- **CLIN**、**Reflexion**、自我演進  
- **多代理**（multi-agent）  
- 具名系統（YieldSentry、ECOFAB…）

我們的強項是 **MCP + deterministic prove engine + KiCad DRC**，提案裡卻強調 **「80–90% 不用 LLM」** — 對工程是優點，對 **Agent 競賽提案** 可能被讀成「比較像編譯器，不像 Agent 研究」。

### C. 隊伍規模

入圍多為 **2–3 人**。單人提案在書院競賽裡可能吃虧（評審對「團隊可完成度」的想像）。

### D. 問題型態：軟體／資料／報告

入圍題目多半是 **分析、知識、報告、輿情、維護決策** — Demo 易用簡報講清楚，不太依賴現場 KiCad 環境。

我們需要 **本機 KiCad、編譯、DRC** — 提案階段評審看不到跑起來的證據時，只能信文字；若文字又像「已完成的 side project」，可能不如「暑假可演進的 Agent 故事」有吸引力。

## 3. Fab-Truth 提案的優勢（沒有消失）

這些在落選後 **仍然成立**：

| 優勢 | 說明 |
|------|------|
| 真實可跑引擎 | `verify-splice`、`verify-tier-c`、MCP — 不是純簡報 |
| 可稽核 | casefile、DRC、gates — 比「LLM 說可以」硬 |
| 與測試驗證精神一致 | 欣銓／測試產業重視 **evidence** — 方向對，但 **用詞沒對到良率／ATE** |
| 差異化 | salvage、splice、bench gates — 入圍隊沒做 |
| 商業敘事 | NPI 沉沒成本、打樣前閘門 — 對 EMS／新創仍有效 |

**落選 ≠ 技術判輸。** 比較像 **賽道敘事與評審偏好** 的問題。

## 4. 可能落選原因（誠實自查）

| 因素 | 嚴重度 | 說明 |
|------|--------|------|
| 半導體關聯太「周邊」 | 高 | 應更貼 **測試治具驗證、良率相關硬體、ATE bring-up 閘門** |
| Agent 敘事偏弱 | 高 | 過度強調 deterministic；少 **規劃—工具—反思—驗證** 迴圈圖 |
| 範例偏 maker | 中 | 澆水、ESP32 salvage — 應改 **探針卡／驗證板／介面治具** 範例 |
| 單人隊 | 中 | 無組員、無學校系所背書 |
| 提案完成度 | 待確認 | 草稿中隊名／組員／簽名為空 — 若送件版本亦如此，會傷害可信度 |
| 已太成熟 | 中 | 「暑修非從零」可能讓評審覺得不像競賽孵化題 |
| 現場 Demo 風險 | 中 | KiCad 依賴在提案階段無法展示 |

## 5. 若再投類似競賽 — 改怎麼包裝

### 5.1 改名與定位（範例）

不要只說「硬體製造準備度」，改貼評審語彙：

> **「Test-Ready Gate Agent」** — 半導體測試介面／驗證板在投產前的 **Agentic 證據閘門**  
> （良率損失來自 **硬體打樣錯誤** 時，與 Yield 敘事接軌）

或：

> **「ATE Bring-up Copilot」** — 連接測試意圖 → BOM／連線 → DRC → **可上電授權** 的多步 Agent

### 5.2 架構圖要長得像入圍隊

```text
目標（測試治具規格）
  → Planner Agent（拆解任務）
  → Tool Agent（hs_compose / hs_inspect_fab / KiCad）
  → Critic / Reflexion（DRC 失敗 → 修正計畫）
  → Verifier（gates + casefile）
  → 輸出：可稽核報告（非僅聊天）
```

後端 **仍可** 是現有 Hardware-Splicer；簡報語言要 **Agent-first**。

### 5.3 範例全部半導體化

| 避免 | 改用 |
|------|------|
| 澆水、RC 玩具 | 簡化驗證板、介面治具、參考設計改版 |
| Salvage junk drawer | 測試廠 **備品模組** 重組 |
| 通用 KiCad | **測試準備度**、**probe 安全**、**上電順序** |

### 5.4 隊伍與合作

- 找元智／半導體相關科系 **1–2 人**（即使分工寫簡報）  
- 或掛 **指導老師／業師** 一句話背書  

### 5.5 提案附件

- **3 分鐘預錄 Demo** 連結（提案階段就給）  
- 一張 **成功 + 失敗** casefile 截圖  
- `PROJECT_PACKAGE.json` 的 **GATES** tab 截圖 — 差異化
